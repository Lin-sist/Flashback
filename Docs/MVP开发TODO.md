# 《时光回序》设计评估与 MVP 开发 TODO

## 1. 设计文档合理性评估

### 1.1 总体判断
这份设计文档的产品方向是成立的，原因有三点：

1. 核心价值明确：不是泛日记工具，而是“节点记录 + 延迟回看 + 可选回信”。
2. 用户场景具体：大学生和年轻人在升学、实习、毕业、求职等节点确实有记录和回望需求。
3. 差异化有抓手：相较普通日记，重点不在“持续写日记”，而在“时间差带来的回看价值”。

如果作为课程项目或简历项目，这个方向是合适的，既有业务表达空间，也有技术实现空间。

### 1.2 当前文档的优点
- 产品定位清楚，目标用户不空泛。
- 核心闭环完整：记录 -> 封存 -> 解锁 -> 回看 -> 可选回信。
- 已经明确“回信不是强制动作”，这个判断正确。
- AI 被定义为辅助角色，而不是主流程主角，这一点合理。
- MVP 思路基本正确，知道要先验证核心价值，而不是一开始做大而全。

### 1.3 当前文档的主要问题

#### 问题 1：MVP 边界还不够硬
文档里同时出现了：
- 用户端
- AI 辅助
- 时间轴
- 提醒机制
- 后台管理
- 数据统计

这些都做会明显超出第一版课程项目的合理范围。

建议：
- 第一阶段只做一个用户端 Web/H5 + 一个极简后台。
- 后台只保留用户列表、记录列表、标签配置，不做复杂报表。

#### 问题 2：AI 功能定义偏多，容易变成伪需求
当前 AI 包含：
- 写作引导
- 内容整理
- 回看问题生成
- 行动建议生成

第一版不需要全做。

建议：
- MVP 只保留 2 个 AI 能力：
  - 写作提示生成
  - 记录内容整理
- 回看问题和行动建议可先写死为规则模板，不必一开始接模型。

#### 问题 3：提醒机制描述过早，且实现成本被低估
“解锁提醒”如果做成短信、邮件、订阅消息，都会抬高实现复杂度。

建议：
- MVP 先只做“登录后可见已解锁记录”和站内提示。
- 如果时间允许，再补邮件提醒。

#### 问题 4：隐私要求提到了，但没有落到实现约束
这是高隐私场景，不能只写“注意隐私”，需要落成具体规则。

建议至少明确：
- 记录默认私密，不支持公开广场。
- 封存后的记录不可编辑。
- 草稿可编辑，已封存不可编辑。
- 后台管理员不能直接修改用户记录正文。
- 敏感字段预留加密方案。

#### 问题 5：缺少核心数据模型和状态流转定义
当前文档偏产品表述，但真正开始开发前，必须先把状态说清楚。

至少要补齐：
- 记录状态：`DRAFT` / `SEALED` / `UNLOCKED`
- 回信状态：`NONE` / `REPLIED`
- AI 处理状态：`NOT_PROCESSED` / `PROCESSED`

#### 问题 6：前后端形态没有收敛
文档中写了“小程序 / H5 / 前后端分离页面 / Web 管理端”，这会让开发目标失焦。

建议：
- MVP 固定为：`Spring Boot + MyBatis + MySQL + Redis + Vue`
- 用户端先做响应式 Web，不做小程序。
- 管理端复用同一套前端工程，走管理路由。

### 1.4 结论
设计方向合理，但当前版本更像“产品蓝图”，还不是“可以直接开工的开发说明书”。

要真正进入编码阶段，必须先收缩为一个明确的 MVP：

`记录当下 -> 设定解锁时间 -> 到期回看 -> 可选一句回信`

AI、提醒、后台、成长分析都只能围绕这个主链路服务，不能反客为主。

## 4. 核心状态设计

### 4.1 记录状态
- `DRAFT`：草稿，可编辑，可删除
- `SEALED`：已封存，未到解锁时间，不可编辑
- `UNLOCKED`：已解锁，可查看，可回信

### 4.2 核心业务规则
- 草稿记录可以反复编辑。
- 记录一旦封存，不允许修改正文和解锁时间。
- 当前时间到达 `unlock_at` 后，记录自动转为 `UNLOCKED`。
- 只有 `UNLOCKED` 状态的记录允许创建回信。
- 每条记录第一版只允许 1 条回信，降低实现复杂度。

## 6. 推荐的开发顺序

不要并行做太多模块，建议按下面顺序推进：

1. 后端骨架
2. 数据库设计
3. 用户登录
4. 记录模块
5. 解锁定时任务
6. 回信模块
7. 前端主流程页面
8. 标签和时间轴
9. AI mock
10. 极简后台

原因：
- 先把主链路打通，再补增强功能。
- AI 和后台都不是第一优先级。
- 如果课程时间有限，做到第 7 步已经可以形成完整演示。

## 7. MVP 验收标准

如果第一版完成后能满足下面 5 条，就算 MVP 成立：

1. 用户可以注册登录，并创建一条记录。
2. 记录可以保存为草稿，也可以封存并设置解锁时间。
3. 到时间后记录能自动解锁并被用户看到。
4. 用户可以在解锁后写一句回信。
5. 前端能完整演示“记录 -> 封存 -> 解锁 -> 回看 -> 回信”。

## 8. 给 GitHub Copilot 的使用建议

不要这样提需求：

`帮我实现一个时光回序项目。`

这样太大，Copilot 很容易生成结构混乱、边界不清的代码。

建议这样拆：

1. 先让 Copilot 生成项目骨架。
2. 再让它单独生成数据库表和实体类。
3. 再逐个生成功能模块：用户、记录、解锁、回信。
4. AI 相关先让它写接口和 mock，不要一开始接真实模型。
5. 每次只给一个明确模块，并附上输入输出和业务规则。
