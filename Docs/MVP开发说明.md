# 《时光回序》MVP 开发说明（V2.0）

本文档用于约束 MVP 第一版的开发范围、业务规则和代码生成边界，目标是让 GitHub Copilot 基于统一约束生成更稳定的代码，减少后期返工。

---

## 1. 文档目标

本说明文档只解决 5 个问题：

1. MVP 第一版到底做什么，不做什么
2. 核心业务对象和状态如何定义
3. 主要接口和页面范围是什么
4. Copilot 生成代码时必须遵守哪些约束
5. 开发顺序和验收标准是什么

本文档不是完整产品文档，不讨论远期功能，只服务于当前 MVP 编码阶段。

---

## 2. MVP 目标

### 2.1 产品目标
实现一个最小可运行的“节点记录与延迟回看”系统，让用户能够完成以下主流程：

`创建记录 -> 保存草稿或封存 -> 到时间自动解锁 -> 回看记录 -> 可选写一句回信`

### 2.2 项目目标
- 可作为课程项目完成演示
- 可作为简历项目展示业务系统设计能力
- 可在后续版本上继续扩展 AI、提醒、多媒体和成长分析能力

### 2.3 核心原则
- 第一版优先打通主链路，不追求功能多
- AI 是辅助能力，不是核心主流程
- 回信是增强体验，不是强制动作
- 隐私和状态流转必须先定义清楚

---

## 3. 技术边界

### 3.1 固定技术栈
- 后端：`Spring Boot 3`
- ORM：`MyBatis`
- 数据库：`MySQL`
- 缓存：`Redis`
- 鉴权：`JWT`
- 用户端：`微信小程序（使用 Uniapp 框架，基于 Vue 3 语法）`
- 管理端：`Vue 3 + Vite + Element Plus`

### 3.2 端形态约束
- 第一版用户端做微信小程序，使用 Uniapp 框架开发
- 第一版管理端做极简 Web 端，使用 Vue 3 + Element Plus
- 第一版不做原生 App、桌面端
- 用户端与管理端分开维护，不共用前端工程

### 3.3 工程约束
- 后端按 `controller / service / mapper / domain / dto / vo` 分层
- 普通用户接口和管理端接口分开
- 所有接口使用统一返回体
- 所有错误使用统一错误码和统一异常处理

## 4. MVP 范围

### 4.1 必须实现

| 分类 | 内容 |
| ---- | ---- |

| 时间模块 | 设置解锁时间、定时解锁、已解锁记录查询、微信订阅消息推送：记录解锁时向用户发送微信订阅消息提醒 |
| 回看模块 | 查看已解锁记录详情 |
| 回信模块 | 写一句回信、查询回信 |
| 标签模块 | 标签列表、记录绑定标签 |

### 4.2 建议实现

| 分类 | 内容 |
| ---- | ---- |
| AI 模块 | 写作提示、内容整理 |
| 后台模块 | 用户列表、记录列表、标签管理、基础统计数量 |

| 分类 | 内容 |

| AI 扩展 | AI 陪聊、长期记忆、复杂对话系统 |

## 6. 枚举与状态流转

这一部分必须固定，否则 Copilot 容易随意扩展状态，导致后期返工。

### 6.1 记录状态 `record.status`
- `DRAFT`：草稿，可编辑，可删除
- `SEALED`：已封存，未解锁，不可编辑
- `UNLOCKED`：已解锁，可查看，可回信

状态流转：

`DRAFT -> SEALED -> UNLOCKED`

### 6.2 回信状态
第一版不单独维护回信状态枚举。

约定：
- `reply` 记录存在，即表示该记录已回信
- 每条记录第一版只允许 1 条回信

### 6.3 记录类型 `record.record_type`
- `FUTURE_LETTER`
- `NODE_RECORD`
- `EMOTION_NOTE`

### 6.4 标签类型 `tag.type`
- `MOOD`
- `TOPIC`

### 6.5 用户状态 `user.status`
- `ENABLED`
- `DISABLED`

## 7. 核心业务规则

### 7.1 用户规则
- 用户必须登录后才能访问自己的记录数据
- 用户只能查看和操作自己的记录、回信
- 管理员接口与普通用户接口分离

### 7.2 记录规则
- 新建记录默认状态为 `DRAFT`
- 草稿状态允许编辑标题、正文、标签、解锁时间
- 草稿状态允许删除
- 记录封存后不允许再编辑标题、正文、标签、解锁时间
- 封存后不允许再删除
- `unlock_at` 必须晚于当前时间
- 记录默认私密，不提供公开可见能力

### 7.3 解锁规则
- 定时任务只处理 `SEALED` 状态且 `unlock_at <= 当前时间` 的记录
- 解锁任务执行后，将记录状态更新为 `UNLOCKED`
- 解锁任务必须幂等，重复执行不能重复解锁同一条记录
- 解锁后写入解锁日志

### 7.4 回信规则
- 只有记录本人才能回信
- 只有 `UNLOCKED` 状态记录允许回信
- 每条记录第一版只允许一条回信
- 回信提交后不可修改、不可删除
- 回信类型第一版可以只支持 `SHORT_REPLY`

### 7.5 标签规则
- 标签为系统预设标签
- 第一版不支持用户自定义标签
- 一条记录可绑定多个标签

### 7.6 AI 规则
- AI 只作为辅助工具，不自动写完整记录
- 第一版只提供“写作提示”和“内容整理”
- AI 能力通过独立服务层封装
- AI 超时、失败或空结果时必须返回兜底文案或空安全结果

### 7.7 微信订阅消息规则
- 用户在封存记录时，系统必须引导用户完成订阅消息授权
- 授权行为由前端调用微信 `wx.requestSubscribeMessage` 接口发起
- 用户授权成功后，后端记录该用户已授权状态
- 用户拒绝授权时，系统降级处理：提示用户到期后需主动打开小程序查看
- 后端在记录解锁时，通过微信服务端接口向已授权用户发送订阅消息
- 订阅消息内容：告知用户“你在[封存日期]写下的记录现在可以打开了”
- 发送失败时记录日志，不影响解锁主流程
- `access_token` 由后端统一管理和刷新，前端不持有
- `openid` 在接入微信登录后获取并存入 `user` 表，不重复请求

## 8. 数据约束建议

这些约束建议在数据库字段、DTO 校验和业务逻辑里同时体现。

### 8.1 用户数据
- `username` 唯一
- `nickname` 非空，长度受限
- `password_hash` 不存明文密码

### 8.2 记录数据
- `title` 可为空
- `content` 非空
- `unlock_at` 在封存时必填
- `record_type` 必须属于约定枚举
- `status` 必须属于约定枚举

### 8.3 回信数据
- `content` 非空
- 回信内容长度需要限制
- 同一 `record_id` 只允许存在一条回信

### 8.4 索引建议
- `record.user_id`
- `record.status`
- `record.unlock_at`
- `reply.record_id`

## 11. Copilot 生成代码约束

这部分是文档的重点，直接决定后续返工量。

### 11.1 通用约束
- 不要生成超出 MVP 范围的功能
- 不要擅自增加新的状态枚举
- 不要擅自增加社交、公开、推荐相关逻辑
- 不要将 AI 能力写成主流程依赖
- 不要把管理端和用户端接口混在一起

### 11.2 后端约束
- 业务逻辑放在 `service`
- `controller` 只做参数接收和响应返回
- DTO 和 VO 分离
- 所有接口走统一返回体
- 所有异常走统一异常处理
- 权限校验必须内置在用户相关业务中
- AI 模型 API 的密钥只能存在后端配置文件中，不得出现在前端代码或请求中
- 所有 AI 请求由前端调用后端 `/api/ai/` 接口，后端负责转发和结果处理

### 11.3 数据约束
- 枚举值使用固定字符串，不要自由发挥
- 记录和回信必须做归属校验
- 解锁任务必须按幂等方式实现
- AI 接口必须支持 mock 和后续替换真实实现

### 11.4 前端约束
- 用户端使用 Uniapp 开发，兼容微信小程序平台
- 页面围绕主流程搭建，不要过度设计复杂交互
- 首页优先展示“新建记录”和“最近解锁”
- 封存页必须包含订阅消息授权引导步骤，授权失败需有降级提示
- 解锁详情页优先强调阅读体验和回信入口
- 前端不直接调用 AI 模型 API，所有 AI 请求必须经过后端中转，避免密钥暴露
- 管理端只做基础表格和表单，不做复杂可视化

## 12. 推荐开发顺序

建议严格按以下顺序推进：

1. 初始化后端工程
2. 设计数据库和建表 SQL
3. 完成注册登录
4. 完成记录模块
5. 完成解锁定时任务
6. 完成回信模块
7. 完成前端主流程页面
8. 完成标签和时间轴
9. 接入 AI mock
10. 完成极简后台

原因：
- 主链路优先
- AI 和后台不是第一优先级
- 做到第 7 步就已经具备完整演示能力

## 13. MVP 验收标准

第一版完成后，至少要满足以下标准：

1. 用户可以注册、登录并获取自己的用户信息
2. 用户可以创建草稿、编辑草稿、删除草稿
3. 用户可以封存记录并设置解锁时间
4. 定时任务可以把到期记录自动更新为 `UNLOCKED`
5. 用户可以查看已解锁记录详情
6. 用户可以对已解锁记录写一句回信
7. 用户可以在时间轴中查看自己的记录
8. 前端可以完整演示主流程

## 14. 给 Copilot 的提问方式

不要一次性提：

`帮我生成整个时光回序项目。`

建议每次只提一个明确任务，并附上本文档中的边界和规则。

### 14.1 示例 1：后端骨架
`请基于 Spring Boot 3 为“时光回序”初始化后端项目，技术栈为 MyBatis + MySQL + Redis + JWT。请按 controller/service/mapper/domain/dto/vo 分层，提供统一返回体、全局异常处理和 JWT 鉴权基础结构，不要生成超出 MVP 的业务代码。`

### 14.2 示例 2：记录模块
`请基于以下规则实现记录模块：record.status 只有 DRAFT、SEALED、UNLOCKED；草稿可编辑和删除；封存后不可编辑和删除；用户只能访问自己的记录。请生成 Controller、Service、Mapper、DTO、VO 和必要的校验逻辑。`

### 14.3 示例 3：解锁任务
`请实现记录解锁定时任务，只处理 SEALED 且 unlock_at 到期的记录，将其改为 UNLOCKED，并保证任务幂等，同时写入解锁日志。`

### 14.4 示例 4：AI 模块
`请设计一个可替换的 AiService，先实现 mock 版本，支持 writing-prompts 和 summarize-record 两个接口，并提供超时失败兜底，不要引入复杂对话能力。`
